<context:annotation-config></context:annotation-config>

Spring容器内部的执行步骤

1.当Spring容器启动的时候 为bean创建对象

2.当Spring容器解析到依赖注入注解解析器的时候

Spring容器会在纳入Spring管理的bean的范围内查找 所有的属性和方法上是否有@Resource注解

3.如果存在该注解

如果该注解的name属性为空（默认为空 不加就是空）

会按照属性的名称和bean的id进行匹配

如果匹配成功 则把bean的对象赋值给它

如果匹配不成功 按照类型进行匹配

如果还匹配不成功 报错

如果该注解的name属性不为空（@Resource(name="...")）

会按照name属性的值和容器中的bean的id进行匹配

成功直接赋值 不成功报错

 

  • @Autowired

按照类型匹配

  • @Qualifier("...")

按照id匹配

Spring容器内部的执行步骤

1.启动Spring容器时 解析context:component-scan配置

2.会在 指定的包及子包下扫描所有的类

3.看类上是否有@Component注解

存在则看注解的value属性

value为空

把类的第一个字母变成小写 作为id值 加入Spring容器

value不为空 @Component("...")

就把value的值作为id 加入Spring容器

XML的形式和注解的形式

XML

缺点：书写麻烦

四、为什么要有IoC和DI 意义？

为了解耦 松耦合

  • Document的栗子

传统结构

Document接口

public interface Document {

public void write();

public void read();

}

WordDocument实现类

DocumentManager

客户端

优点 在DocumentManager中有一个引用Document是一个接口

缺点 如果要切换文档 客户端的代码还要变

Spring容器

客户端

配置文件

<bean id="documentManager" class="....DocumentManager">

<property name="document" ref="wordDocument"/>

</bean>

<bean id="wordDocument" class="....WordDocument"></bean>

<bean id="pdfDocument" class="....PdfDocument"></bean>

DocumentManager中的document是由Spring容器赋值的 配置文件由程序员配置

客户端无需关注Document是由哪个类来实现的

从Spring容器中把DocumentManager提取出来就行了

就完全地面向接口编程了 松耦合
